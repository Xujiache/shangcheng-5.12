## 1. 项目特性规范

### 1.1 产品形态
- **项目类型**：多端商城系统（O2O / B2B2C 混合形态）
- **业务模式**：厂家 → 门店代理 → 客户三级供应链 + 平台撮合
- **三端五屏**：
  - 用户端 · 微信小程序
  - 商家后台 · APP（厂家 + 门店共用，按角色显隐）
  - 商家后台 · PC（运营批处理）
  - 平台管理 · APP（移动审核、应急）
  - 平台管理 · PC（运营核心）

### 1.2 角色体系
| 角色 | 来源 | 主要权益 |
|------|------|----------|
| 平台管理员 | 内部 | 全部审核 + 配置 + 数据 |
| 厂家 | 商家入驻申请 → 平台审核 | 商品发布、被代理、推广 |
| 门店 | 商家入驻申请或客户升级 | 代理厂家商品、自营销售、店铺装修 |
| 普通客户 | 微信授权登录 | 浏览 / 下单 / 评价 / 售后 |
| 分佣客户 | 普通客户 + 推广分佣开通 | 推广海报、佣金提现 |

### 1.3 关键业务约束
1. **价格分级显示**（核心约束，渗透每个商品展示场景）
   - 未登录 → 不显示价格
   - 普通客户 → 仅零售价
   - 门店（已授权） → 可见批发价
   - 平台会员 → 会员价
2. **会员体系**：试用 30 天 → 月费 ¥99 / 年费 ¥899（基础包）
3. **广告推送套餐**：基础 ¥299/月、进阶 ¥999/月、旗舰 ¥9,800/年 + 增值单项
4. **选品广场**：平台推送商品/厂家 → 门店一键代理（含建议加价区间、佣金）
5. **平台功能开关**：平台可下发控制商家 APP 端按钮 / 菜单显隐，支持灰度

### 4.2 验收标准（初稿）
1. **视觉**：每屏与原型布局结构、信息密度、关键元素位置 1:1；用商业级视觉语言替换手稿风

3. **数据**：前端层 mock 数据驱动完整 CRUD；后端接入后无缝切换
4. **响应式**：APP 端适配 iPhone SE → iPhone 15 Pro Max 宽度；PC 端适配 1280 → 1920
5. **企业级**：可上线生产，含部署文档、运维手册、监控告警

## 5. 智能决策（基于行业知识自答）

### 5.1 仓库结构 ✅ pnpm monorepo（最佳实践）

```
jiujiu-mall/
├── packages/
│   ├── user-mp/         # 用户端微信小程序（uni-app）
│   ├── merchant-app/    # 商家 APP（uni-app → Android+iOS+H5）
│   ├── merchant-pc/     # 商家 PC（Vue3 + Vite + Element Plus）
│   ├── platform-app/    # 平台 APP（uni-app → Android+iOS+H5）
│   ├── platform-pc/     # 平台 PC（Vue3 + Vite + Element Plus）
│   ├── shared/          # 共享：design tokens、UI 原子、Mock 数据、API 类型、工具
│   └── server/          # 后端服务（待 5.5 拍板）
├── docs/                # 6A 流程产物 + 接口文档
├── deploy/              # docker-compose、k8s、CI/CD
├── 原型图/              # 原型源（已存在）
├── pnpm-workspace.yaml
├── package.json
├── .env.example
└── README.md
```

**选型理由**：
- 五端共用 design tokens、API 类型、Mock 数据，monorepo 是唯一不重复造轮子的方案
- pnpm 比 npm/yarn 在 hoist、磁盘占用、CI 速度都更优
- uni-app 官方推荐 monorepo 接入方式

### 5.3 mock 策略 ✅

| 端 | 工具 | 持久化 |
|----|------|--------|

**切换机制**：环境变量 `VITE_USE_MOCK=true` / `process.env.USE_MOCK=true`，一键切换 mock / 真实后端。

## 6. 待最终拍板（4 处关键决策）

下列 4 处必须在进入 CONSENSUS / DESIGN 前敲定，避免架构返工。

### Q1. 后端技术栈
- **方案 A**：Node + NestJS + TypeORM + PostgreSQL（前后端语言统一，开发效率高）
- **方案 B**：Java + SpringBoot + MyBatis Plus + MySQL（企业级金标准，国内最主流）
- **方案 C**：Go + Gin/Kratos + GORM + PostgreSQL（性能极致、并发强、运维轻）

### Q2. 小程序平台覆盖
- **方案 A**：仅微信小程序（原型明确写"微信"，最简单）
- **方案 B**：微信 + 支付宝 + 抖音（uni-app 一码多端，扩展性强）
- **方案 C**：全平台（微信、支付宝、抖音、百度、QQ、快手）

### Q3. APP 形态
- **方案 A**：uni-app 编译原生包（.apk / .ipa，可上架应用市场）
- **方案 B**：纯 H5 套壳 + 桌面图标（开发快、无审核，但无原生能力）
- **方案 C**：A + B 双形态并行（H5 内嵌运营预览，原生包正式上架）

### Q4. 部署目标
- **方案 A**：阿里云（OSS + ACK + RDS + Redis 云数据库 + SLS 日志）
- **方案 B**：腾讯云（COS + TKE + 云数据库 + 云函数）
- **方案 C**：自托管（自建机房 / VPS + Docker Compose + MinIO + 自管 MySQL/Redis）
- **方案 D**：先 Docker Compose 跑通本地，部署目标后期再决定
